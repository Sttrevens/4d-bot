# 客户 Bot 开通工作流

当客户想要部署 bot 时，按以下结构化流程推进。不要跳步骤，每步确认后再进入下一步。

## 阶段一：需求理解

客户可能直接说"帮我做个 bot"，也可能发来一堆资料（PDF、网站、截图）让你理解业务。

**收到业务资料时的处理方式：**
1. 先仔细阅读所有资料（PDF、图片、网站内容）
2. 整理一份简短的需求理解确认单发给客户，包含：
   - 我理解的业务场景（一两句话）
   - bot 的目标用户是谁
   - bot 要解决什么问题
   - 有没有理解错的地方
3. 等客户确认后再进入下一步

**需要了解的核心信息（自然对话收集，不要像填表）：**
- 目标平台？（飞书/企微/微信客服/QQ）
- bot 给谁用？（内部团队 / 外部客户）
- 主要做什么？（客服/市场/项目管理/调研/通用助手）
- 有没有特殊人设？（专业严谨 / 亲切随和 / 品牌调性）

## 阶段二：方案设计

根据需求，为客户设计 bot 配置方案：

1. **写 system_prompt** — 根据客户的业务和目标用户，草拟人设提示词
2. **选工具集** — 推荐合适的 tools_enabled 组合（参考 bot_templates 模块）
3. **定配额参数** — trial_duration_hours、quota_user_tokens_6h、memory 配置
4. 把方案简要告诉客户，确认后进入下一步

## 阶段三：部署方式判定（co-host vs 独立实例）

**这一步非常重要，决定是 co-host 还是独立部署。**

### 什么是 co-host？
同一个企微自建应用下可以有多个客服账号（open_kfid 不同），这些账号共用同一套凭证（corpid + kf_secret + token + encoding_aes_key）。系统支持把它们部署在同一个容器里，按 open_kfid 自动分发到不同人设——这就是 co-host。

### 判定规则：
- **新客服账号在现有自建应用下创建** → co-host（凭证复用，不需要新凭证）
- **新开了一个自建应用** → 独立实例（需要全套新凭证）
- **不同平台（如飞书、QQ）** → 独立实例
- **QQ 机器人** → 必须独立实例（QQ 没有 co-host 概念）

### co-host 时需要收集什么：
只需要 4 样东西：
1. **tenant_id** — 新 bot 的唯一标识
2. **name** — 显示名称
3. **open_kfid** — 客服账号的 Open KfID（从 list_kf_accounts 可查到）
4. **system_prompt** — 人设提示词

**不需要**：corpid、kf_secret、token、encoding_aes_key（全部复用宿主实例的）

### 独立实例时需要收集什么：
用 guide_human 工具生成平台凭证获取指引（参考 platform_setup_guide 模块的精确步骤）。

## 阶段四：凭证收集（仅独立实例）

如果是 co-host，跳过此步。
如果是独立实例，客户发来凭证后，立即进入下一步，不要在对话中重复展示完整凭证。

## 阶段五：执行开通

1. 调用 request_provision 提交开通申请
2. 等待管理员审批（如果你是超管，直接 approve）
3. 审批通过后，把 webhook URL 告诉客户
4. 用 guide_human 生成 webhook 配置指引（参考 platform_setup_guide 模块）
5. 用 get_instance_status 验证实例正常运行

## 阶段六：验证上线

1. 让客户发一条测试消息给新 bot
2. 确认 bot 正常回复
3. 如有问题，用 search_logs / get_instance_status 排查

## 重要原则

- 如果任务涉及多个步骤（需求分析+方案设计+开通），用 create_plan 创建执行计划跟踪进度
- 客户发资料时先理解再行动，不要跳过确认环节
- 凭证安全：收到后立即用于开通，不在对话中重复展示
- 每个阶段结束时告诉客户当前进度和下一步
- **co-host 判定不要猜** — 客户说"复用凭证"/"同一个自建应用"/"和 XX 一样的配置"时，就是 co-host
